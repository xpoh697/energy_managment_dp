## 1. Обзор системы
Система предназначена для автоматизации работы гибридного солнечного инвертора с целью максимизации прибыли от арбитража электроэнергии (продажа по высокому тарифу, покупка по низкому/отрицательному) при безусловном обеспечении энергетической безопасности дома. Система интегрирует прогнозы солнечной генерации, исторические профили потребления и рыночные цены для построения оптимального 48-часового плана работы.

## 1. Лимиты разряда (Конституция)
1. **Единый Гейткипер (Unified Gatekeeper)**: Основной защитный механизм, имеющий два режима работы:
2. **Turbo Mode (04:00 – 10:00)**:
    - Лимит: `MinSOC (emergency) + 2%`.
    - Пользовательский лимит (`ai_discharge_limit_soc`) и прогноз потребления дома **игнорируются**.
    - Цель: Максимальный выжим энергии перед рассветом.
3. **Safe Mode (10:00 – 04:00)**:
    - Применяется логика **Обратного Резервирования (Reverse Bridging)**.
    - **Целевой уровень выхода (`T_end`)**: рассчитывается на последний час продажи в ночном блоке (до 04:00).
    - `T_end = max(User_Limit, MinSOC + Buffer + Дом_после_продажи_до_рассвета)`.
      * *Примечание*: `User_Limit` (`ai_discharge_limit_soc`) — это порог, ниже которого запрещена коммерческая продажа (т.е. на конец продажи SOC не может быть меньше этого лимита).
      * *Примечание*: `MinSOC + Buffer` — это минимальный уровень выживания дома на рассвете.
    - **Текущий пол (Floor)**: `T_end + Дом_от_сейчас_до_конца_продаж`.
    - Цель: Гарантировать соблюдение лимита выживания в самой низкой точке ночи, учитывая потребление дома в паузах между продажами, при этом не давая продавать ниже `User_Limit`.
4. **Stop Sell**: Продажа из АКБ запрещена, если цена ниже `price_stop_sell`.
5. **Saturation Bypass**: Если симуляция предсказывает перезаряд (SOC=100%) в ближайшие 24ч, лимит безопасности может быть временно снижен до `MinSOC`.
6. **Sunrise Safety Check**: Финальный этап планирования. После распределения бюджета продаж запускается контрольная 48-часовая симуляция. Если она показывает, что в час рассвета (08:00) уровень заряда упадет ниже `MinSOC + 2%`, мощность продаж принудительно сокращается до тех пор, пока лимит выживания на утро не будет соблюден.

## 2. Сбор и обработка данных

### 2.0. Интеграция с Home Assistant (Сбор данных)

Для интеграции с Home Assistant (HA) система использует встроенное ядро `HomeAssistant core` и обращается к стейт-машине напрямую.

**1. Механизм получения состояния объектов**:
- Чтение текущих показаний датчиков (мощность генерации, нагрузка дома, SOC АКБ, сетевые перетоки) выполняется через прямой вызов стейт-машины HA:
  `state_obj = hass.states.get(entity_id)`
- Это исключает дополнительные сетевые задержки и гарантирует доступ к самым актуальным значениям в реальном времени.

**2. Умная нормализация и парсинг единиц (`get_kwh_val`)**:
Любое строковое значение состояния, полученное из HA, проходит обязательную очистку и приведение к единым расчетным единицам (кВт/кВтч) с помощью функции `get_kwh_val`:
- **Очистка от мусора**: Состояния `"unknown"`, `"unavailable"`, `None` или `"None"` автоматически отфильтровываются и возвращают `None`.
- **Поддержка любых форматов**: Числа преобразуются из строкового формата с заменой запятых на точки (поддержка европейской и русской локалей через `normalize_float`).
- **Автоматическое масштабирование**: В зависимости от `unit_of_measurement` объекта:
    - Если единица `Wh` или `W` — значение делится на `1000.0` для перевода в `кВт` / `кВтч`.
    - Если единица `MWh` или `MW` — значение умножается на `1000.0` для перевода в `кВт` / `кВтч`.
    - Для стандартных `kWh` и `kW` значение возвращается без изменений.
- Это предотвращает фатальные ошибки в расчете баланса энергии из-за несоответствия масштабов датчиков разных интеграций инвертора (например, Solcast выдает в кВтч, а инвертор — в Втч).

**3. Подписка на события изменений**:
Для мгновенного реагирования на изменения ключевых параметров (SOC, текущие нагрузки) система подписывается на шину событий HA с помощью `async_track_state_change_event`. Это позволяет пересчитывать баланс мощностей и обновлять состояние инвертора в реальном времени при резких изменениях нагрузок дома или генерации.

### 2.0.1. Диагностический доступ к настройкам и истории HA при поиске багов

В процессе автономной отладки, локального тестирования и поиска скрытых багов (например, неверно определившихся лимитов SOC или «слепых» зон в симуляциях) AI-агент или разработчик использует прямые методы извлечения данных из файлов конфигурации и базы данных Home Assistant через Python.

#### 1. Чтение реальных настроек интеграции из Config Entries
Поскольку настройки интеграции (MinSOC, лимиты, буферы, имена датчиков) хранятся в защищенном внутреннем реестре HA, их прямое чтение выполняется через парсинг файла `.storage/core.config_entries`.

Пример Python-скрипта отладки для извлечения настроек:
```python
import json
import os

def get_management_settings(config_dir: str):
    path = os.path.join(config_dir, ".storage", "core.config_entries")
    if not os.path.exists(path):
        return None
    with open(path, "r", encoding="utf-8") as f:
        data = json.load(f)
    for entry in data.get("data", {}).get("entries", []):
        if entry.get("domain") == "energy_management_dp":
            return {
                "entry_id": entry.get("entry_id"),
                "title": entry.get("title"),
                "data": entry.get("data", {}),
                "options": entry.get("options", {})
            }
    return None
```
Это позволяет мгновенно понять, какие датчики (`CONF_CONSUMPTION_SENSORS`, `CONF_BATTERY_SOC`) и лимиты сейчас активны в системе без запуска самого Home Assistant.

#### 2. Извлечение истории датчиков напрямую из SQLite (`home-assistant_v2.db`)
Для детального анализа аномалий в поведении инвертора и проверки симуляций AI-агент обращается к локальной базе данных SQLite. Это позволяет выгрузить точные исторические ряды SOC, нагрузок и генерации за любой период.

Пример Python-кода для выборки истории:
```python
import sqlite3
import json

def query_sensor_history(db_path: str, entity_id: str, limit: int = 100):
    conn = sqlite3.connect(db_path)
    cursor = conn.cursor()
    
    # Запрос исторических состояний и временных меток
    query = """
        SELECT s.state, s.last_updated_ts
        FROM states s
        JOIN states_meta m ON s.metadata_id = m.metadata_id
        WHERE m.entity_id = ?
        ORDER BY s.last_updated_ts DESC
        LIMIT ?
    """
    cursor.execute(query, (entity_id, limit))
    results = cursor.fetchall()
    conn.close()
    return [(r[0], r[1]) for r in results]
```
Данный подход используется диагностическими скриптами (например, `analyze_db.py`, `probe_anomalies.py`) для выявления сбоев датчиков, нулевых чтений профилей и некорректного КПД силовой части.

#### 3. Автономный запуск симуляции (Dry-Run / Репликация бага)
Для воссоздания бага агент создает изолированный скрипт-песочницу (например, `scratch.py`), который:
1. Инициализирует облегченную версию `StrategyEngine` с mock-менеджером `self.manager`.
2. Наполняет объект настроек (`self.manager.get_setting`) реальными значениями из `.storage/core.config_entries`.
3. Загружает исторические профили из `home-assistant_v2.db` за аварийные сутки.
4. Прогоняет симуляцию `run_soc_simulation()` по шагам и выводит промежуточные логи в консоль.
5. Это позволяет мгновенно обнаружить зацикливания или некорректный порядок вычисления режимов без перезапуска промышленного сервера HA.

### 2.1. Генерация (Solar Forecast)
Алгоритм прогнозирования генерации является многоуровневым и сочетает внешние прогнозы с глубоким анализом исторической точности.

**1. Источники данных**:
- `f_today`, `f_tom`: Общий прогноз выработки (кВтч) от Solcast на сегодня и завтра.
- `dist_today`, `dist_tom`: Почасовое распределение весов (кривая генерации) от Solcast.
- *Fallback*: При отсутствии почасового прогноза от Solcast используется форма исторического профиля генерации за последние 14 дней.

**2. Почасовая коррекция точности (`h_acc`)**:
Для каждого часа симуляции вычисляется коэффициент исторической точности на основе последних 14 дней:
- `h_acc = Average(Actual_Gen / Forecast_Gen)` для данного часа.
- Коэффициент ограничен диапазоном `[0.2, 2.0]`.
- **Исключение данных**: Часы, в которых генерация была принудительно ограничена (Curtailment), **не учитываются** в расчете точности. Ограничение детектируется, если:
    - Инвертор был в режиме `stop_sale` при SOC > 90%.
    - Цены на покупку были отрицательными (выгоднее было не генерировать/не брать солнце).
    - Инвертор был в режиме `no_pv_sale_no_bat`.

**3. Динамическое смешивание (`blended_coeff`)**:
В течение дня система пересчитывает доверие к прогнозу в реальном времени:
- `Today_Performance = Actual_Today / Expected_So_Far` (насколько мы реально эффективнее прогноза с начала дня).
- `Progress = Sum(Weights_Past) / Sum(Weights_Total)` (какая часть "солнечного веса" дня уже прошла).
- `blended_coeff = (Today_Performance * Progress) + (1.0 * (1-Progress))`.
*Логика*: Утром (Progress ≈ 0) система доверяет базовому прогнозу (`1.0`). К вечеру (Progress ≈ 1) система полностью полагается на реальную производительность дня.
- Коэффициент ограничен диапазоном `[0.3, 1.5]`.

**4. Финальный расчет мощности в симуляции**:
`P_gen[h] = (Weight[h] / Sum_Remaining_Weights) * Remaining_Forecast * blended_coeff * h_acc[h]`

**5. Ночной зажим (Night Clamp)**:
Для предотвращения превращения технических шумов датчиков в "фантомную генерацию" ночью, прогноз принудительно обнуляется для часов с 20:00 до 08:00, если исторический профиль для этого часа близок к нулю.

### 2.2. Потребление (Consumption Profiles)
Система ведет два типа профилей:
- **Consumption Total**: Полное потребление дома. Используется для оценки общих затрат.
- **Consumption Base**: "Чистое" фоновое потребление (Total минус управляемые нагрузки). На основе этого профиля строится стратегия "выживания" АКБ и расчет резерва.

**Исключение данных из профилей (Data Cleaning)**:
Чтобы профили не искажались аномалиями, система игнорирует или корректирует следующие данные:
- **Аномальные скачки**: Изменения счетчика энергии более 100 кВтч за один замер (считается ошибкой датчика).
- **Управляемые нагрузки**: Прыжки потребления отдельных приборов более 2 кВтч за замер.
- **Режим "Healed" (Лечение профиля)**: 
    - **Генерация**: Если в данный час генерация была принудительно ограничена (Curtailment), в профиль записывается **прогнозное значение** (Forecast), чтобы система не "привыкала" к заниженным показателям.
    - **Потребление**: Если в данный час цена была отрицательной (<= 0.05) или система принудительно покупала энергию по стратегии (`buy`), реальное потребление может быть искусственно завышено. В этом случае в профиль записывается **среднестатистическое** значение для этого часа вместо фактического.
- **Переходные сезоны**: В марте, апреле, сентябре и октябре окно обучения сокращается с 14 до 7 дней для быстрой адаптации к изменению светового дня.

## 3. Движок симуляции (SOC Simulation)

Симуляция — это ядро системы, предсказывающее состояние заряда АКБ (SOC) на 24-48 часов вперед.

### 3.1. Параметры расчета
- **Шаг**: 1 час.
- **КПД (Efficiency) — Динамический расчет**:
    Система не использует фиксированный коэффициент. КПД рассчитывается динамически на основе истории за последние 14 дней:
    - **Фильтрация**: Учитываются только замеры с мощностью генерации > 1.0 кВт. Это исключает влияние "холостого хода" инвертора на статистику и дает реальный КПД силовой части при рабочих нагрузках арбитража.
    - **Формула**: `eff_ratio = (Gen_Total - Losses_Total) / Gen_Total`.
    - **Границы**: Коэффициент ограничен диапазоном `[0.85, 0.99]`.
    - **Default**: При отсутствии данных принимается `0.95` (95%).
- **Применение в симуляции**:
    - **Заряд**: `P_batt = P_available * eff_coeff`.
    - **Разряд**: `P_batt = P_needed / max(0.85, eff_coeff)`.
- **Разделение потерь**: 
    - Конверсионные потери учитываются через `eff_coeff`.
    - Постоянные потери (Self-consumption/Standby) учитываются отдельно через добавление почасовой нагрузки `idle_p` (из исторического профиля `losses`) в общий баланс дома.
- **Баланс мощности**: `P_delta = (P_solar * blended_coeff) - (P_load_adj + idle_p) + P_strategy`.
    - **P_load_adj**: В режиме покупки (`allow_discharge=False`) нагрузка дома `P_load` принимается равной **0**, так как она покрывается сетью напрямую (байпас). В остальных режимах используется полное прогнозное значение.
- **Лимиты**: Симуляция учитывает емкость АКБ (кВтч) и не позволяет SOC выходить за пределы 0-100% (физически) и `ai_discharge_limit_soc` (логически).

## 4. Алгоритмы стратегий
### 4.1. Арбитраж (Sell Strategy)
Алгоритм не просто ищет выгодную цену, а управляет "пулом энергии" на 48 часов вперед.

**1. Поиск пиков (Peak Discovery)**:
Система находит все часы, где цена продажи выше лимита (`sell_limit`) или ожидается избыток солнца.

**2. Группировка в пулы (Energy Pooling)**:
Часы объединяются в один пул, если между ними **нет солнечной подзарядки** (часы 08:00 - 18:00).
- *Пример*: Пик сегодня в 19:00 и пик завтра в 06:00 — это **один пул**. У них один и тот же запас энергии в АКБ.
- *Пример*: Пик сегодня в 19:00 и пик завтра в 19:00 — это **разные пулы**. Между ними АКБ зарядится от солнца.

**3. Приоритезация по цене (Greedy Price Priority) — v11.8.543**:
Внутри одного пула часы сортируются по **цене (убывание)**. Энергия из общего бюджета (`global_budget`) распределяется в режиме «Lion Eats First»: сначала самый дорогой час забирает `min(max_batt_p, rem_budget)`, затем следующий по цене, и так далее. Это гарантирует максимальную концентрацию мощности в самых выгодных окнах, даже если они идут позже по времени.

**4. Оптимизатор двойного цикла (Double Cycle Optimizer)**:
Если между двумя пулами (например, утро и вечер) ожидается солнце, но симуляция показывает, что его **не хватит** для полной зарядки АКБ до 100%:
- Система сравнивает средние цены Пула 1 и Пула 2.
- Если цена во втором пуле выше (минимум на 0.05), система **блокирует** продажу в первом пуле, сохраняя энергию для более дорогого окна.

**5. Рекурсивная коррекция (Recursive Fix)**:
После распределения энергии запускается финальная 48-часовая симуляция. Если она показывает отклонение SOC на рассвете от целевого значения:
- Система рекурсивно подстраивает мощность продажи (в обе стороны) до тех пор, пока прогноз на рассвете не совпадет с целевым порогом (`min_soc_bat + soc_buffer`).

6. **Алгоритм итеративной аллокации (Budget-Based Iterative Allocator) — v11.9.689**:
    Финальный план продаж строится через управление **глобальным бюджетом энергии** с безусловным приоритетом безопасности (TS 1.6):
    1. **Initial Guess**: Бюджет рассчитывается как весь заряд выше `active_safety_floor`.
    2. **Baseline Simulation & Sunrise Guard**: Перед итерациями запускается симуляция с нулевыми продажами.
        - **Критический барьер**: Если даже в Baseline SOC на рассвете падает ниже `MinSOC + 2.0%`, бюджет продаж принудительно обнуляется. Продажи запрещены (Sunrise Safety Block).
    3. **Distribution**: Если база безопасна, бюджет распределяется по часам согласно приоритету цены.
    4. **Bidirectional Refinement & Natural Deficit Bypass**:
        - **Deficit Detection**: Считается "добавленный дефицит" относительно Baseline.
        - **Budget Adjustment**: Если продажи вызвали падение SOC ниже пола, бюджет сокращается с демпфированием.
        - **Ceasefire Rule**: Если бюджет близок к нулю, а дефицит всё равно есть (естественный дефицит), система останавливает корректировку.
    5. **Convergence**: Цикл повторяется (до 20 раз), обеспечивая максимальную прибыль при 100% соблюдении Конституции системы.

**7. Горизонты анализа и стабильность "Sunset-to-Sunset" — v11.9.685**:
    Система оперирует двумя временными горизонтами для обеспечения баланса между долгосрочной выгодой и краткосрочной стабильностью:
    - **Горизонт планирования (48 часов)**: Система всегда анализирует цены и прогноз генерации на двое суток вперед. Это необходимо для работы *Double Cycle Optimizer* (решение о переносе продажи на завтра) и предварительного поиска выгодных окон.
    - **Окно валидации дефицитов (Sunset-to-Sunset)**: Несмотря на 48-часовой обзор, проверка на дефицит SOC и корректировка бюджета продаж (`target_budget_ac`) выполняются строго в окне до **следующего реального заката**.
    - **Логика**: Если в итеративном аллокаторе обнаруживается просадка SOC ниже пола *после* следующего заката, она **игнорируется**. Это гарантирует, что текущий "ночной цикл" продаж защищен от влияния потенциально неточных прогнозов на вторые сутки.

**8. Учет КПД и лимитов**:
- Бюджет считается в **DC-энергии**, а команды на инвертор выдаются в **AC** (`AC = DC * Eff`).
- Проверка лимитов (`is_solar_surplus`, `max_batt_p`) встроена в этап распределения.

### 3.2. Режимы симуляции
1. **Baseline Simulation**: Расчет SOC при условии отсутствия активных команд управления (только солнце и дом). Позволяет найти "Естественный SOC" (Natural SOC).
2. **Strategic Simulation**: Расчет с учетом планируемых продаж или покупок. Используется для проверки безопасности стратегии.
3. **Recursive Deficit Fix**: Если финальная симуляция показывает отклонение SOC ниже порога выживания, система рекурсивно поднимает/опускает защитный лимит и пересчитывает мощность продажи.

## 4. Стратегии управления
### 4.2. Покупка (Buy Strategy)
Алгоритм покупки управляет заполнением АКБ из сети, минимизируя затраты и обеспечивая готовность к дефицитным периодам.

**1. Ценовой фильтр и арбитраж (Price filtering & Arbitrage)**:
Алгоритм разрешает покупку из сети в двух случаях:
- **Ниже лимита**: Текущая цена покупки (`price_buy`) ниже установленного пользователем порога (`price_buy_limit`).
- **Выгодный арбитраж**: Покупка сейчас по цене `P1` выгодна для продажи позже по цене `P2`. Условие выгоды: `(P2 * Eff) - P1 - DegCost >= Threshold`.
  - `Eff`: Текущий КПД системы.
  - `DegCost`: Стоимость деградации АКБ на 1 кВтч цикла.
  - `Threshold`: Порог прибыли (`arbitrage_profit_threshold`).

**2. Логика отрицательных цен (Negative Price Logic)**:
Если в 48-часовом окне обнаружены часы с ценой <= 0:
- **Целевой SOC**: Для часов с отрицательной ценой `target_soc` устанавливается на **100%**.
- **Блокировка PV**: Блокировка заряда от солнца (режим `no_pv_sale_no_bat`), чтобы зарезервировать емкость для «бесплатной» энергии из сети.
- **Приоритет**: Покупка при отрицательной цене имеет безусловный приоритет.

### 4.2.1. Алгоритм выживания и мостовой зарядки (Survival & Bridge)
Это критический подраздел стратегии покупки, отвечающий за предотвращение глубокого разряда АКБ в периоды отсутствия солнца и высоких цен.

**1. Детекция дефицита (12-цикловая симуляция)**:
Система запускает итеративный процесс (до 12 циклов) для поиска «точек невозврата»:
- **Единый триггер (Deadline)**: `SOC < MinSOC + 5.0%`. Если симуляция предсказывает падение ниже этого уровня в любой час до рассвета — инициируется режим выживания.
- **Учет нагрузки**: Симуляция автоматически учитывает фоновое потребление дома (`consumption_base`), поэтому дедлайн является интегральным показателем выживаемости.
- **Горизонт**: Сканирование идет до рассвета (`morning_h_abs`).

**2. Выбор окна дозарядки (Decision Logic)**:
После нахождения часа дедлайна система выбирает окно в интервале `[Anchor, Deadline]`:
- **Якорь (Anchor)**: Конец последней ручной продажи (`sale_pv_bat`) или текущий час. Закупка планируется строго **после** всех ручных разрядов.
- **Case A (Global Cheapest)**: Если самый дешевый час до рассвета наступает **раньше** дедлайна и **позже** якоря — выбираем его.
- **Case B (Bridge Charging)**: Если глобальный минимум недоступен (за якорем или после дедлайна) — ищем лучший час внутри доступного интервала.
- **Fallback**: Если подходящих окон нет (например, ручная продажа стоит впритык к аварии), используется `cheapest_global` как принудительная мера.

**3. Целевой уровень (Targeting)**:
- **Уровень дозарядки**: Зарядка всегда происходит до уровня `Survival_Floor + 5.0%`.
- **Логика**: Создается 10%-й «гистерезис» безопасности (от -5% до +5%). Это предотвращает частые включения инвертора и гарантирует, что купленной энергии хватит на несколько часов жизни дома, даже если прогноз нагрузки слегка превышен.

**4. Статус активности (Active State)**:
Стратегия считается `active`, если текущий час запланирован для покупки И:
- Расчетная мощность покупки (`power_needed`) > **0.05 кВт**.
- **ИЛИ** текущая цена **отрицательная** (статус `active` сохраняется для индикации окна даже при полной АКБ).

**5. Диагностика и прозрачность**:
В атрибут `buy_debug` выводится детализация: 
`Цена: [P_now] | Лучшая продажа позже: [P_sell] | Лучшая покупка позже: [P_buy] | Выгода арб: [G] (Порог: [T])`.
Дополнительно выводятся статусы `[Отрицательная цена]` или `Ожидание отрицательных цен`.

**6. Жадная CC/CV-осознанная аллокация (v11.6.425)**:
При распределении мощности по отрицательным или дешевым часам аллокатор имитирует рост SOC и корректирует доступную мощность каждого часа согласно функции `get_cc_cv_ratio`. Это гарантирует, что система зарезервирует достаточно времени и часов для завершения зарядки до 100% малым током (фаза CV), предотвращая недобор заряда в финальном прогнозе.

**7. Стабилизация и Hourly Locking (v11.6.516)**:
Для обеспечения стабильной работы инвертора и исключения «дребезга» команд в течение часа применяется механизм **High-Water Mark Anchor**:
- **Фиксация в начале часа**: При наступлении нового часа параметры стратегии (Power, Target SOC, Amps) фиксируются и передаются инвертору.
- **Разрешение на рост**: Если в течение часа расчетный бюджет растет (появилась возможность заряжать большей мощностью), якорь обновляется.
- **Запрет на падение**: Если расчетная мощность падает, инвертор продолжает работать на ранее зафиксированном максимуме до конца часа.
- **Target SOC Floor**: В режиме ожидания покупки (Patience Mode) целевой SOC принудительно подпирается снизу текущим уровнем заряда (`Target_SOC = max(Current_SOC, Sim_Target)`). Это предотвращает ложные команды на разряд АКБ под нагрузкой дома до начала дешевого окна.

## 5. Режимы работы инвертора (State Machine)

| Режим | Описание | Условия активации |
| :--- | :--- | :--- |
| **`buy`** | Активная покупка | Работает AI Buy. Дом питается от сети параллельно зарядке (байпас, не разряжает АКБ). |
| **`no_pv_sale_no_bat`** | Ожидание "минуса" | Блокирует заряд АКБ от солнца и экспорт. Ждем отрицательных цен для зарядки из сети. (не заряжает АКБ)|
| **`sale_pv_no_bat`** | Только солнце | Продаем излишки PV, но НЕ заряжаем АКБ (бережем место для ночного "минуса"). Только до 13:00 (настраиваемо). |
| **`sale_pv_bat`** | Арбитраж (AI) | Активная продажа излишков солнца и накопленной энергии АКБ по команде стратегии. |
| **`stop_sale`** | Запрет экспорта | Все солнце идет только на дом и заряд АКБ. Цена ниже `price_stop_sell`. |
| **`sale_pv`** | Стандарт | Излишки солнца продаются, АКБ заряжается. Обычный дневной режим. |
| **`bat_emergency`** | Аварийный режим | SOC ниже критического уровня (`ai_discharge_limit_soc`). Блокировка разряда до добора энергии. |

### 5.1 Приоритеты режимов инвертора (Inverter Mode Ladder) — v11.9.691
Для исключения конфликтов между эвристиками и стратегиями ИИ, выбор режима в `_get_mode_at` (sensor.py) строго следует иерархии:
1. **bat_emergency** (Приоритет 1: Выживание) — Активируется при критическом SOC без излишков генерации.
2. **buy** (Приоритет 2: Покупка) — Работа по расписанию StrategyBuy или отрицательной цене.
3. **sale_pv_bat** (Приоритет 3: Арбитраж AI) — Активная продажа энергии из АКБ по команде StrategySell. **Перебивает утренние эвристики**.
4. **sale_pv_no_bat** (Приоритет 4: Heuristic Morning Mode) — Продажа только солнца при высокой цене для экономии места в АКБ.
5. **no_pv_sale_no_bat** — Ожидание отрицательных цен.
6. **stop_sale** — Цена ниже порога остановки.
7. **sale_pv** — Стандартная работа (питание дома + заряд АКБ от солнца).

## 6. Защитные механизмы (Safety Blocks)
1. **Sunrise Guard (Защита рассвета)**: Механизм динамического резервирования энергии для гарантированного выживания дома до рассвета.
    - **Safe Mode (10:00 - 04:00)**:
      - Основная цель: Обеспечить дом энергией до утра и соблюсти лимит пользователя.
      - Использует «обратную проекцию» — к целевому уровню на рассвете добавляется прогнозный расход дома от конца продажи до восхода.
      - Логика **Reverse Bridging** учитывает нагрузку дома внутри ночного пула продаж.
    - **Turbo Mode (04:00 - 10:00)**:
      - Цель: Максимальная прибыль. Лимит: `MinSOC + 2%`.
    - **Sunrise Safety Check**: Безусловный программный барьер. Система выполняет финальную «прогонку» всего плана через движок симуляции. Если на конец ночного цикла (к рассвету) прогнозируется просадка ниже `MinSOC + 2.0%`, стратегия признается небезопасной, и бюджет продаж рекурсивно урезается.
2. **Double Cycle Optimizer**: Блокировка продаж вечером, если завтра ожидается более выгодная цена и АКБ не успеет зарядиться.
3. **Smart Deficit Throttling**: Плавное ограничение продаж при дефиците солнца.

## 7. Архитектура файлов
- **`const.py`**: Константы, настройки по умолчанию и текущая версия.
- **`sensor.py`**: 
    - `EnergyProfileManager`: Центральный координатор. Хранит данные, управляет профилями, детектирует "Healed" режимы.
    - `InverterOperationModeSensor`: Логика выбора режима (`_get_mode_at`).
    - Описание всех сенсоров (профили,Savings, ROI, Anomaly).
- **`strategy.py`**: Математическое ядро.
    - `StrategyEngine`: Расчет стратегий BUY/SELL.
    - `run_soc_simulation`: Универсальный движок прогнозирования SOC.
    - `get_budget_and_permissions`: Расчет солнечного бюджета и лимитов разряда.
- **`binary_sensor.py`, `number.py`, `switch.py`**: Интерфейсы управления настройками системы.

## 8. Развертывание и Синхронизация (Deployment & Synchronization)

Для обеспечения корректной работы системы и актуальности исполняемого кода в среде Home Assistant, необходимо строго соблюдать следующие правила развертывания:

**1. Целевой адрес на сервере**:
Обязательный бамп версии перед деплоем в const.py и в manifest.json 
Все файлы интеграции (из папки `custom_components/energy_management_dp/`) должны синхронизироваться по адресу:
`\\192.168.100.5\config\custom_components\energy_management_dp`

**2. Принудительная очистка кэша**:
После каждой синхронизации файлов на сервере **ОБЯЗАТЕЛЬНО** удалять папку `__pycache__` по целевому адресу:
`\\192.168.100.5\config\custom_components\energy_management_dp\__pycache__`
*Логика*: Home Assistant может использовать скомпилированные байт-код файлы (`.pyc`), игнорируя изменения в исходном коде (`.py`), что приводит к исполнению старых версий логики.

**3. Процедура обновления**:
1. Синхронизация файлов (рекомендуется `robocopy` с флагом `/XO`).
2. Удаление `__pycache__` на сервере.
3. Перезагрузка интеграции или всего Home Assistant.

**4. Резервное копирование базы данных (Debugging)**:
При необходимости глубокой отладки профилей потребления/генерации или при риске повреждения данных, ИИ-ассистент может запросить у пользователя свежий бэкап базы данных.
Бэкап должен быть размещен пользователем по адресу:
`\\192.168.100.5\config\energy_management_dp_backup.json`
