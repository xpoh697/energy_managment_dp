## 1. Обзор системы (Архитектура v12.0)
Система автоматизирует работу гибридного инвертора для максимизации прибыли от арбитража электроэнергии при обеспечении энергобезопасности дома. 

**Ключевое отличие v12.0**: Переход от фрагментированной логики к **Глобальному Плану Диспетчеризации (Global Dispatch Plan)**. План является единственным источником правды (Single Source of Truth) для всей системы. Все расчеты режимов, мощностей и прогнозов SOC централизованы в 48-часовом реестре.

## 2. Глобальный План Диспетчеризации (Global Dispatch Plan)

### 2.1. Структура Реестра
План состоит из 48 объектов `GlobalSlot`, каждый из которых представляет один час начиная с текущего:
- **Входные данные**: Цены (Buy/Sell), Прогноз генерации (Raw/Adj), Прогноз потребления (Base/Total).
- **Решение (Decision)**: Режим инвертора (Mode), Целевая мощность (Power AC), Целевой SOC (Target SOC), Причина (Reason).
- **Симуляция (Projections)**: SOC на начало/конец часа, чистый поток АКБ.
- **Отладка**: Полные дампы стратегий Buy/Sell для конкретного часа.

### 2.2. Цикл Обновления (Orchestration)
План обновляется каждую **1 минуту** (метод `async_update_global_plan` в `EnergyProfileManager`):
1. **Сбор предложений**: Стратегии `StrategyBuy` и `StrategySell` генерируют свои «пожелания» (proposals).
2. **Слой логики (Logic Engine)**: Единый движок `EnergyLogicEngine` проходит по всем 48 часам и выбирает финальный режим, учитывая иерархию приоритетов.
3. **Единая Симуляция**: Выполняется сквозной расчет SOC на 48 часов, устраняющий дрейф данных между разными датчиками.
4. **Стабилизация 0-го часа**: Для текущего часа (Slot 0) применяется логика реального времени (Power Stabilization), учитывающая текущий SOC и минутную динамику.

## 3. Лимиты разряда (Конституция)
1. **Единый Гейткипер (Unified Gatekeeper)**: Основной защитный механизм, интегрированный в `EnergyLogicEngine`.
2. **Turbo Mode (04:00 – 10:00)**: Лимит: `MinSOC (emergency) + 2%`. Цель: Максимальный выжим перед рассветом.
3. **Safe Mode (10:00 – 04:00)**: Логика **Reverse Bridging**. Ограничивает коммерческую продажу лимитом пользователя (`ai_discharge_limit_soc`) на конец продаж, при этом гарантируя соблюдение лимита выживания (`MinSOC + Buffer`) в самой низкой точке ночи (на рассвете).
4. **Sunrise Safety Guard**: Безусловный программный барьер. Если 48-часовой Глобальный План показывает падение SOC на рассвете ниже `MinSOC + 2.0%`, бюджет продаж в плане принудительно сокращается до безопасного уровня.

## 4. Сбор и обработка данных (Inputs)

### 4.0. Интеграция с Home Assistant (Сбор данных)

Для интеграции с Home Assistant (HA) система использует встроенное ядро `HomeAssistant core` и обращается к стейт-машине напрямую.

**1. Механизм получения состояния объектов**:
- Чтение текущих показаний датчиков (мощность генерации, нагрузка дома, SOC АКБ, сетевые перетоки) выполняется через прямой вызов стейт-машины HA:
  `state_obj = hass.states.get(entity_id)`
- Это исключает дополнительные сетевые задержки и гарантирует доступ к самым актуальным значениям в реальном времени.

**2. Умная нормализация и парсинг единиц (`get_kwh_val`)**:
Любое строковое значение состояния, полученное из HA, проходит обязательную очистку и приведение к единым расчетным единицам (кВт/кВтч) с помощью функции `get_kwh_val`:
- **Очистка от мусора**: Состояния `"unknown"`, `"unavailable"`, `None` или `"None"` автоматически отфильтровываются и возвращают `None`.
- **Поддержка любых форматов**: Числа преобразуются из строкового формата с заменой запятых на точки (поддержка европейской и русской локалей через `normalize_float`).
- **Автоматическое масштабирование**: В зависимости от `unit_of_measurement` объекта:
    - Если единица `Wh` или `W` — значение делится на `1000.0` для перевода в `кВт` / `кВтч`.
    - Если единица `MWh` или `MW` — значение умножается на `1000.0` для перевода в `кВт` / `кВтч`.
    - Для стандартных `kWh` и `kW` значение возвращается без изменений.
- Это предотвращает фатальные ошибки в расчете баланса энергии из-за несоответствия масштабов датчиков разных интеграций инвертора (например, Solcast выдает в кВтч, а инвертор — в Втч).

**3. Подписка на события изменений**:
Для мгновенного реагирования на изменения ключевых параметров (SOC, текущие нагрузки) система подписывается на шину событий HA с помощью `async_track_state_change_event`. Это позволяет пересчитывать баланс мощностей и обновлять состояние инвертора в реальном времени при резких изменениях нагрузок дома или генерации.

### 4.0.1. Диагностический доступ к настройкам и истории HA при поиске багов

В процессе автономной отладки, локального тестирования и поиска скрытых багов (например, неверно определившихся лимитов SOC или «слепых» зон в симуляциях) AI-агент или разработчик использует прямые методы извлечения данных из файлов конфигурации и базы данных Home Assistant через Python.

#### 1. Чтение реальных настроек интеграции из Config Entries
Поскольку настройки интеграции (MinSOC, лимиты, буферы, имена датчиков) хранятся в защищенном внутреннем реестре HA, их прямое чтение выполняется через парсинг файла `.storage/core.config_entries`.

Пример Python-скрипта отладки для извлечения настроек:
```python
import json
import os

def get_management_settings(config_dir: str):
    path = os.path.join(config_dir, ".storage", "core.config_entries")
    if not os.path.exists(path):
        return None
    with open(path, "r", encoding="utf-8") as f:
        data = json.load(f)
    for entry in data.get("data", {}).get("entries", []):
        if entry.get("domain") == "energy_management_dp":
            return {
                "entry_id": entry.get("entry_id"),
                "title": entry.get("title"),
                "data": entry.get("data", {}),
                "options": entry.get("options", {})
            }
    return None
```
Это позволяет мгновенно понять, какие датчики (`CONF_CONSUMPTION_SENSORS`, `CONF_BATTERY_SOC`) и лимиты сейчас активны в системе без запуска самого Home Assistant.

#### 2. Извлечение истории датчиков напрямую из SQLite (`home-assistant_v2.db`)
Для детального анализа аномалий в поведении инвертора и проверки симуляций AI-агент обращается к локальной базе данных SQLite. Это позволяет выгрузить точные исторические ряды SOC, нагрузок и генерации за любой период.

Пример Python-кода для выборки истории:
```python
import sqlite3
import json

def query_sensor_history(db_path: str, entity_id: str, limit: int = 100):
    conn = sqlite3.connect(db_path)
    cursor = conn.cursor()
    
    # Запрос исторических состояний и временных меток
    query = """
        SELECT s.state, s.last_updated_ts
        FROM states s
        JOIN states_meta m ON s.metadata_id = m.metadata_id
        WHERE m.entity_id = ?
        ORDER BY s.last_updated_ts DESC
        LIMIT ?
    """
    cursor.execute(query, (entity_id, limit))
    results = cursor.fetchall()
    conn.close()
    return [(r[0], r[1]) for r in results]
```
Данный подход используется диагностическими скриптами (например, `analyze_db.py`, `probe_anomalies.py`) для выявления сбоев датчиков, нулевых чтений профилей и некорректного КПД силовой части.

#### 3. Автономный запуск симуляции (Dry-Run / Репликация бага)
Для воссоздания бага агент создает изолированный скрипт-песочницу (например, `scratch.py`), который:
1. Инициализирует облегченную версию `StrategyEngine` с mock-менеджером `self.manager`.
2. Наполняет объект настроек (`self.manager.get_setting`) реальными значениями из `.storage/core.config_entries`.
3. Загружает исторические профили из `home-assistant_v2.db` за аварийные сутки.
4. Прогоняет симуляцию `run_soc_simulation()` по шагам и выводит промежуточные логи в консоль.
5. Это позволяет мгновенно обнаружить зацикливания или некорректный порядок вычисления режимов без перезапуска промышленного сервера HA.

### 4.1. Генерация (Solar Forecast)
- **Динамическое смешивание (`blended_coeff`)**: Смешивает реальную производительность дня с прогнозом Solcast.
- **Почасовая коррекция (`h_acc`)**: Учитывает историческую точность прогноза для каждого часа за последние 14 дней.
- **Исключение Curtailment**: Часы с принудительным ограничением генерации не используются для обучения точности.

### 4.2. Потребление (Consumption Profiles)
- **Consumption Total**: Для финансовых расчетов.
- **Consumption Base**: Для стратегии выживания.
- **Data Cleaning**: Игнорирование аномалий > 100 кВтч и "лечение" профиля прогнозными значениями в периоды ограничений.

## 5. Движок Логики и Симуляции (EnergyLogicEngine)

Все вычисления перенесены в `dispatch_plan.py` для обеспечения чистоты и тестируемости.

### 5.1. Выбор Режима (Mode Selection)
Иерархия приоритетов (сверху вниз):
1. **Manual Overrides**: Ручные часовые команды пользователя (Priority 1).
2. **Emergency/Survival**: Защита от глубокого разряда (Priority 2).
3. **AI Buy**: Команды на закупку от StrategyBuy (Priority 3).
4. **AI Sell/Arbitrage**: Команды на продажу от StrategySell (Priority 4).
5. **Heuristics**: Утренние эвристики (sale_pv_no_bat) и ожидание отрицательных цен (no_pv_sale_no_bat).
6. **Default**: Стандартная работа (`sale_pv`).

### 5.2. Расчет Мощности (Power Calculation)
- **DC/AC Efficiency**: Используется динамический КПД [0.85-0.99].
- **Real-time Stabilization**: В текущем часе мощность корректируется в зависимости от того, сколько времени осталось до конца часа и какой SOC нужно достичь.
- **CC/CV Awareness**: Симуляция учитывает замедление заряда при SOC > 90%.

## 6. Алгоритмы стратегий (Proposals)

Стратегии больше не управляют инвертором напрямую. Они подают данные в Глобальный План.

### 6.1. Арбитраж (Sell Strategy)
- **Energy Pooling**: Группировка часов в блоки между солнечными подзарядками.
- **Lion Eats First**: Самые дорогие часы внутри пула забирают бюджет первыми.
- **Double Cycle Optimizer**: Перенос продажи на завтра, если это выгоднее и АКБ не успеет зарядиться.

### 6.2. Покупка (Buy Strategy)
- **Arbitrage Profitability**: `(P_sell * Eff) - P_buy - DegCost >= Threshold`.
- **Negative Price Logic**: Безусловный приоритет зарядки и блокировка PV для освобождения емкости АКБ.
- **Bridge Charging**: Дозарядка до уровня `Survival + 5%`, если симуляция предсказывает дефицит до рассвета.

## 7. Отображение и UI (Frontend)

Сенсоры Home Assistant теперь являются «прокси-интерфейсами» Глобального Плана.
- **Inverter Operation Mode Sensor**:
    - `native_value`: Текущий режим из Slot 0.
    - `planned_modes_24h`: Список режимов на сутки из Глобального Плана.
    - `hourly_data`: Полный набор данных (цены, режимы, прогноз SOC, генерация, нагрузка, причины) для отрисовки графиков.
- **Parity Policy**: Запрещено удалять существующие атрибуты. Любые новые данные (напр. прогноз генерации в часе) добавляются как новые ключи в `hourly_data`.
